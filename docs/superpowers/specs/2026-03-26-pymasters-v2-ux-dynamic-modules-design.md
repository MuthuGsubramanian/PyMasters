## 1. Classroom UX Overhaul

### 1.1 Container & Layout

- **Container:** Expand from `max-w-5xl` (1024px) to `max-w-screen-xl` (1280px) with `px-8` padding
- **Intro phase layout:** Change from `grid-cols-5` (2/5 + 3/5) to `grid-cols-3` (1/3 story + 2/3 animation)
  - Left column (1/3): Story cards, concept maps
  - Right column (2/3): Code animations, variable boxes, terminal output, data structures
- **Practice phase:** Widen from `max-w-2xl` (672px) to `max-w-4xl` (896px)
- **Feedback phase:** Widen from `max-w-2xl` to `max-w-4xl`
- **Chat bar:** Remains fixed at bottom, full-width within container
- **Responsive:** On screens < 1024px, stack vertically — story on top, animation below

### 1.2 Structured Learning Cards

Story cards and concept cards must render structured content instead of plain text blocks:

- **Section headings** — h2/h3 hierarchy for topic organization
- **Bullet points** — Key takeaways, step-by-step explanations
- **Tables** — Comparisons (e.g., `for` vs `while`, mutable vs immutable types)
- **Inline code snippets** — Highlighted code within explanations
- **Key Concept callout boxes** — Highlighted boxes for critical concepts
- **Numbered steps** — For procedural explanations

All content rendered via ReactMarkdown with remark-gfm for full GitHub-flavored markdown support (already in place).

### 1.3 Files Affected

- `frontend/src/pages/Classroom.jsx` — Container widths, grid layout, phase widths
- `frontend/src/components/animations/StoryCard.jsx` — Structured content rendering
- `frontend/src/components/animations/ConceptMap.jsx` — Potential layout adjustments
- Lesson JSON files — Story content updated to use structured markdown

## 3. Dynamic Module Generation System

### 3.1 Triggers

**Explicit (user request):**
- User types in chat: "teach me about decorators" or "I want to learn prompt engineering"
- Vaathiyaar acknowledges and queues the module for generation
- Tagged: `"trigger": "user_request", "trigger_detail": "user asked about decorators"`

**AI-inferred:**
- **Struggle detection:** User fails same concept twice → deep-dive module queued
- **Interest inference:** User asks 3+ chat questions about a topic outside current path → intro module queued
- **Mastery gap:** User has high mastery in prerequisite but skipped a dependent topic → suggested module
- Tagged: `"trigger": "ai_recommended", "trigger_detail": "struggled with closures in functions module"`

### 3.2 Multi-Stage Generation Pipeline

| Stage | Description | Input | Output | Validation |
|-------|-------------|-------|--------|------------|
| 1. Outline | Generate learning objectives and subtopics | Topic + user profile + mastery map | JSON: objectives, subtopics, lesson count (1 initially, more adaptive) | Objectives are measurable, topic is valid |
| 2. Narrative | Generate structured story content | Outline + user language/culture + learning style | Markdown story per lesson with sections, points, tables, metaphors | Markdown structure check, language correctness |
| 3. Animation | Select and compose animation templates | Narrative + animation template library + concept type | Animation sequence JSON per lesson | JSON schema validation, all primitives exist in the renderer |
| 4. Challenges | Generate practice code and tests | Narrative + animation + skill level | Practice challenges with code, expected output, hints, test cases | Code executes successfully in sandbox, tests pass with correct solution |
| 5. Assembly & Validation | Combine all stages into final lesson JSON | All above outputs | Complete lesson JSON matching pre-built format | Full schema validation, dry-run animation render check |

### 3.3 Adaptive Depth

- First lesson generates immediately (high priority in the job queue)
- After user completes first lesson, Vaathiyaar evaluates performance:
  - Scored well + showed interest → generate next lesson automatically
  - Struggled → generate a simpler reinforcement lesson
  - Moved on → don't generate more unless user returns
- Maximum depth: 5 lessons per dynamically generated module (prevents runaway generation)
- Each follow-up lesson generation triggered by lesson completion event

### 3.4 Background Job System

```
Table: module_generation_jobs
- id: UUID
- user_id: INTEGER (FK users)
- topic: TEXT
- trigger: TEXT ("user_request" | "ai_recommended")
- trigger_detail: TEXT
- status: TEXT ("queued" | "stage_1" | "stage_2" | "stage_3" | "stage_4" | "stage_5" | "completed" | "failed")
- current_stage_data: JSON (intermediate outputs)
- result_lesson_id: TEXT (FK lessons, set on completion)
- error_message: TEXT (set on failure)
- priority: INTEGER (1=high, user request; 2=medium, AI inferred; 3=low, follow-up)
- created_at: TIMESTAMP
- updated_at: TIMESTAMP
```

- Background worker polls for queued jobs, processes one stage at a time
- On failure at any stage, retry up to 2 times before marking as failed
- On completion, store the lesson JSON and trigger notification

### 3.5 Animation Template Library

Composable animation patterns mapped to concept types:

| Concept Type | Primary Template | Common Compositions |
|-------------|-----------------|---------------------|
| Loop/iteration | CodeStepper + VariableBox | + TerminalOutput, + FlowArrow |
| Data structure | DataStructure | + VariableBox, + ComparisonPanel |
| Function call | CodeStepper + MemoryStack | + FlowArrow, + TerminalOutput |
| Comparison | ComparisonPanel | + CodeStepper (side-by-side) |
| Algorithm | CodeStepper + FlowArrow | + DataStructure, + VariableBox |
| Neural network | ConceptMap + FlowArrow | + DataStructure (weight matrices) |
| I/O operation | CodeStepper + TerminalOutput | + VariableBox |
| Concept explanation | StoryCard + ConceptMap | + ComparisonPanel |

Templates stored as JSON patterns in `backend/animation_templates/` directory. Each template defines:
- Required props with placeholder markers
- Optional props with defaults
- Valid composition rules (which primitives can follow which)

## 4. Notification System

### 4.1 In-App Notifications

- Bell icon in top navigation bar with unread count badge
- Dropdown panel showing recent notifications (last 20)
- Notification types:
  - `module_ready` — "Your lesson on {topic} is ready!"
  - `ai_recommendation` — "Vaathiyaar suggests: {topic}"
  - `achievement` — "Achievement unlocked: {name}"
  - `streak_reminder` — "Keep your streak alive! Practice today."
- Click navigates directly to the relevant lesson
- Mark as read on click, "Mark all as read" button

### 4.2 Email Notifications

- Triggered on `module_ready` and `ai_recommendation` events
- Vaathiyaar-branded HTML email template
- Content: module title, trigger reason, direct link to lesson
- Sent via SMTP or SendGrid API
- Respect `notification_preferences` — opt-in/opt-out per type

### 4.3 WhatsApp Notifications

- Using existing `whatsapp` field in users table
- Integration via Twilio WhatsApp Business API or Meta Cloud API
- Short, conversational messages in Vaathiyaar's voice
- Sent in user's preferred language
- Example (Tamil): "வணக்கம்! Decorators பாடம் தயார். வந்து படிங்க! 🎓"
- Example (English): "Hey! Your lesson on Decorators is ready. Come learn! 🎓"
- Respect quiet hours and notification preferences

### 4.4 Database Schema

```sql
-- Notifications
CREATE TABLE notifications (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NOT NULL REFERENCES users(id),
    type TEXT NOT NULL,  -- 'module_ready', 'ai_recommendation', 'achievement', 'streak_reminder'
    title TEXT NOT NULL,
    message TEXT NOT NULL,
    link TEXT,  -- relative URL to navigate to
    metadata JSON,  -- additional context (lesson_id, trigger, etc.)
    read INTEGER DEFAULT 0,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Notification delivery tracking (for email/WhatsApp)
CREATE TABLE notification_deliveries (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    notification_id INTEGER NOT NULL REFERENCES notifications(id),
    channel TEXT NOT NULL,  -- 'email', 'whatsapp'
    status TEXT NOT NULL,  -- 'pending', 'sent', 'delivered', 'failed'
    sent_at TIMESTAMP,
    error_message TEXT
);

-- User notification preferences
CREATE TABLE notification_preferences (
    user_id INTEGER NOT NULL REFERENCES users(id),
    channel TEXT NOT NULL,  -- 'in_app', 'email', 'whatsapp'
    type TEXT NOT NULL,  -- notification type or 'all'
    enabled INTEGER DEFAULT 1,
    quiet_hours_start TEXT,  -- HH:MM format
    quiet_hours_end TEXT,    -- HH:MM format
    PRIMARY KEY (user_id, channel, type)
);
```

## 5. Fine-Tuned Vaathiyaar Model

### 5.1 Approach

Single fine-tuned base model with dynamic per-request context injection.

### 5.2 Fine-Tuning Scope

- **Teaching patterns** — Explanation style, metaphor usage, scaffolding difficulty, progressive hints
- **Animation composition** — Mapping concepts to animation primitives and valid compositions
- **Structured JSON output** — Consistent response format: `{ message, language, phase, animation, profile_update }`
- **Domain knowledge** — Python (basics → advanced), AI/ML, deep learning, neural networks
- **Multilingual teaching** — Cultural metaphors per language (Tamil proverbs, cricket analogies, chai-making, etc.)

### 5.3 Training Data Sources

1. All pre-built lesson JSONs (the comprehensive curriculum)
2. Curated teaching interactions (high-quality Vaathiyaar responses)
3. Animation template library with concept-to-animation mappings
4. Existing `training_data.py` collection system — continues collecting interaction data
5. User feedback signals (implicit: completion rates, explicit: ratings if added)

### 5.4 Dynamic Context Injection

System prompt assembled per-request from user data:

```
[Vaathiyaar Identity — baked into fine-tune]
[Student Profile]
  - Skill level: {beginner|intermediate|advanced}
  - Learning style: {visual|reading|hands-on}
  - Preferred language: {en|ta|te|...}
  - Goals: {career_switch|upskill|student|...}
[Mastery Map]
  - Strong: {list of mastered topics}
  - Weak: {list of struggling topics}
  - Current: {active topic}
[Recent Signals — last 10]
  - {signal_type}: {value} at {timestamp}
[Current Context]
  - Lesson: {lesson_id}, Phase: {phase}
  - Module type: {pre-built|generated}
```

### 5.5 Model Hosting

- Continue with Ollama Cloud API (OpenAI-compatible format)
- Fine-tuned model replaces base Qwen 3.5
- Fallback to base model if fine-tuned model is unavailable
- Model versioning: `vaathiyaar-v1`, `vaathiyaar-v2`, etc.

### 5.6 Continuous Improvement

- Learning signals feed into training data collection
- Periodic re-fine-tuning as dataset grows (monthly or after significant data milestones)
- A/B testing between model versions (track completion rates, mastery improvements)

## 6. New API Endpoints

### 6.1 Module Generation

```
POST   /api/modules/request          — User requests a topic for module generation
       Body: { user_id, topic, context? }
       Response: { job_id, status: "queued", estimated_time }

GET    /api/modules/status/{job_id}  — Check generation job progress
       Response: { job_id, status, current_stage, progress_pct }

GET    /api/modules/generated/{user_id} — List all generated modules for user
       Response: { modules: [{ id, topic, trigger, status, created_at }] }
```

### 6.2 Notifications

```
GET    /api/notifications                    — Fetch user notifications (paginated)
       Query: ?unread_only=true&limit=20&offset=0
       Response: { notifications: [...], unread_count }

PUT    /api/notifications/{id}/read          — Mark notification as read
PATCH  /api/notifications/read-all           — Mark all as read

GET    /api/notifications/preferences        — Get notification preferences
PUT    /api/notifications/preferences        — Update preferences
       Body: { channel, type, enabled, quiet_hours_start?, quiet_hours_end? }
```

### 6.3 Trigger Engine

```
POST   /api/modules/trigger-check           — Called internally after learning signals
       Body: { user_id, signal_type, topic, value }
       Response: { triggered: bool, job_id?, reason? }
```

## 7. Database Changes Summary

### Lesson Storage Migration

Currently, pre-built lessons are JSON files on disk (`backend/lessons/*.json`). This needs to evolve:

- **Pre-built lessons** remain as JSON files on disk, loaded into memory at startup. This keeps them version-controlled and easy to edit.
- **Generated modules** are stored in a `generated_lessons` database table (SQLite JSON column for the full lesson payload). This allows per-user content without filesystem sprawl.
- **Unified lesson API:** The classroom API reads from both sources transparently — pre-built from disk, generated from DB. The frontend doesn't know the difference.

### New Tables
- `module_generation_jobs` — Background job tracking for module generation
- `generated_lessons` — Dynamically generated lesson content (full JSON payload per lesson)
- `notifications` — In-app notification storage
- `notification_deliveries` — Email/WhatsApp delivery tracking
- `notification_preferences` — Per-user, per-channel notification settings

### Modified Tables
- No schema changes to existing tables. Generated lesson metadata (trigger, user_id, track) lives in `generated_lessons` and `module_generation_jobs` tables.
